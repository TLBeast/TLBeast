## Fixed-size vs variable-size

**Paging** makes free-space management easy — memory is carved into fixed-size units. The OS keeps a list of free units and hands out the first entry.

**Variable-sized allocation** (`malloc()` / `free()`) is harder because of **external fragmentation**.

## External fragmentation

Imagine the heap looks like this:

```
[ free: 10 ] [ allocated ] [ free: 10 ]
addr 0                    addr 20
```

Free list:

```
head → [addr: 0, len: 10] → [addr: 20, len: 10] → NULL
```

Total free = 20 bytes — but any request **greater than 10** fails. There's no single contiguous chunk large enough.

## Splitting

If a request is smaller than a free chunk, the allocator finds a chunk that fits and **splits** it:

1. First part → returned to the caller
2. Second part → stays on the free list

## Coalescing

When memory is freed, merge adjacent free blocks into one larger contiguous chunk — **coalescing**.

## Headers

Allocators stash metadata (like block size) in a **header** next to each chunk.

### Embedding the free list in free space

Start with a 4096-byte (4KB) region:

```
head → [size: 4088, next: 0]
```

The header itself takes 8 bytes → 4088 left for the rest of the chunk.

### After allocating 100 bytes

Split the big free chunk:

```
[size: 100] → ptr to the 100-byte region   (allocated)
head → [size: 3980, next: 0]               (remaining free)
```
