## Why segmentation?

With a single base/bounds pair, unused space inside the address space is still reserved in physical memory — **internal fragmentation**.

**Segmentation** is a slight generalization of base and bounds. A **segment** is a contiguous portion of the address space of a particular length. The OS places logically different segments (code, stack, heap) into **different parts of physical memory**.

## Each segment gets its own base + bounds

Example configuration:

| Segment | Base | Size |
|---|---|---|
| Code | 32K | 2K |
| Heap | 34K | 2K |
| Stack | 28K | 2K |

### Example 1 — code

Fetch virtual address 100:

- Physical = `100 + 32KB = 32868`
- Bounds check: `100 < 2KB` ✓

### Example 2 — heap

Virtual address 4200. Heap starts at 4KB in the virtual address space, so the true offset is:

```
4200 − 4096 = 104
Physical = 104 + 34KB = 34920
```

Accessing something like 7KB (outside the allocated heap range) → **segmentation fault**.

## How do we know which segment?

**Explicit:** top few bits of the virtual address name the segment (`00` = code, `01` = heap, …). Remaining bits are the offset.

**Implicit:** hardware infers from how the address is used — PC fetches → code; stack/base pointer → stack; everything else → heap.

## Stack grows backwards

The stack grows **negatively**, so you need an extra bit for growth direction. Bounds check: absolute size of the negative offset must be less than the segment size.

## Sharing & protection

Segments can be **shared** across address spaces to save memory — e.g. read-only code with **protection bits**.

## Granularity

- **Coarse-grained** — a few large segments
- **Fine-grained** — many smaller segments

On a context switch, **segment registers** must be saved and restored.

## Trade-off

Segmentation helps with **internal fragmentation**, but now you have to deal with **external fragmentation** — free physical memory gets chopped into awkward holes.
