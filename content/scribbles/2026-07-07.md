## Virtualization ≠ virtual memory

**Virtualization** is one thing. **Virtual memory** is a different beast.

First: understand the **virtual address** → which leads to **address spaces** → which leads to **ease of use**, **isolation**, and **protection**.

## From simple machines to time sharing

Early OS developers had it easy — not much user expectation, and early machines didn't provide much abstraction.

Then the **time-sharing era** was born:

1. Run one process for a while, give it full access to its memory
2. Save its state to disk
3. Load another process's state
4. Crude sharing — but **too slow** as memory grows

Saving/restoring registers isn't the problem. Writing the **entire memory to disk** is. So we leave processes in memory while switching.

## The address space abstraction

With time sharing, **protection** matters — but we also need an easy-to-use abstraction of physical memory: the **address space**.

A process's address space contains:

- **Code** — the program's instructions
- **Stack** — tracks the function call chain, local variables, return values
- **Heap** — dynamically allocated, user-managed memory (`malloc()`)

### Sample 16KB address space

```
┌─────────────┐  0 KB
│    Code     │
├─────────────┤  1 KB
│    Heap  ↓  │  (grows up)
├─────────────┤
│   (free)    │
├─────────────┤
│  Stack  ↑   │  (grows down)
└─────────────┘  16 KB
```

The heap and stack **grow and shrink** within the address space.

**Note:** this looks clean in a single-threaded process. With **multiple threads** coexisting in one address space, the layout gets messier.

## Virtual → physical

Virtualization needs virtual memory.

If Process A loads at **virtual address 0**, that might map to **physical address 320KB**.

The OS and hardware handle the translation — each process gets the illusion of its own private memory starting at 0.
