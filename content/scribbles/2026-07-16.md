## The problem

A TLB caches **virtual → physical** translations for the **currently running** process. Those translations aren't valid for other processes.

When switching from process to process, hardware and/or the OS must make sure the about-to-run process **doesn't use translations from the previous one**.

### Example

Process P1: VPN 10 → PFN 100, valid = 1  
Process P2 (after context switch): VPN 10 → PFN 170, valid = 1

Same VPN, different PFN — and the hardware **can't tell which entry belongs to which process**.

## Approach 1: flush the TLB

On a context switch, **flush** the TLB.

- Software-managed TLB → explicit privileged instruction
- Hardware-managed TLB → flush when the **page-table base register** changes

Flush sets all **valid bits to 0** and clears the contents. The new process never accidentally hits a wrong translation.

**Cost:** every time a process runs, you get TLB misses again as code/data pages are touched.

## Approach 2: address space identifiers (ASIDs)

Add an **ASID** field so identical VPN translations from different processes can be differentiated — no full flush required.

## Cache replacement

To install a new TLB entry, an old one has to go. Goal: **minimize miss rate / maximize hit rate**.

- **LRU** (least-recently used) — takes advantage of locality in the memory-reference stream
- **Random** — simpler alternative
