**OS = software that manages the hardware.**

Today's focus: **virtualization**.

## Physical → virtual

The OS takes a physical resource and makes it virtual. That's where the term **virtual machine** comes from.

**What do we mean by physical to virtual?**

Think about watching a movie. When I was a kid, my dad would go to a Redbox at Walmart and buy a CD. We'd kick that CD to the big boards and watch the movie from home.

Time passed and Netflix came out of nowhere. **That's virtual.**

I don't need a CD — I can stream five tabs, five movies at once. All I did was split the physical computer into VMs.

**Boom** — now we know what a **hypervisor** is: the specialized software layer that tricks each VM into believing it has its own physical hardware.

## Using the OS standardly

So the OS is there — but how do we use it in a standard way?

**APIs. Syscalls. Standard library.**

We share the CPU amongst programs → the OS as **resource manager**.

## One CPU vs many

Say I run a program that periodically prints a character on a single CPU/processor:

```
"A" ... "A" ... "A"
```

Now say I run four CPUs at once. Process IDs get listed along with letters A–D running down the output.

That leads us to the **which/when** policy discussion — who runs, and when?
