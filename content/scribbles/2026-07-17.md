## The problem: huge linear page tables

32-bit address space (`2³²` bytes), 4KB pages (`2¹²`), 4-byte PTEs:

- Virtual pages = `2³² / 2¹²` = **~1 million**
- Page table size = 1M × 4 bytes = **4MB per process**

100 active processes → **hundreds of megabytes** just for page tables.

## Band-aid: bigger pages

Larger pages → fewer entries → smaller tables.

**Downside:** more **internal fragmentation**.

## Hybrid: paging + segmentation

Instead of one giant page table for the whole address space, give **each logical segment** its own page table (code, heap, stack).

### MMU registers

- **Base** → physical address of that segment's page table
- **Bounds / limit** → end of that page table

### Virtual address layout

| Bits | Meaning |
|---|---|
| Top 2 | Which segment |
| VPN | Page within the segment |
| Offset | Byte within the page |

## Still not perfect

Even with the hybrid approach, you still get:

- Page table waste
- **External fragmentation**

**Next up:** multi-level page tables.
