## Multicore

The source is the **multicore processor** — multiple CPU cores packed onto a single chip.

Computer architects struggle to make a single CPU faster without excessive power use, so the industry moved to multicore.

**The problem:** a typical application only uses one CPU. Adding more CPUs doesn't make it faster unless the app is rewritten to run in parallel using **threads** — multithreaded apps.

**The OS question:** how do we schedule jobs on multiple CPUs?

## Hardware caches

The fundamental difference between single-CPU and multi-CPU hardware.

**Caches** are small, fast memories that hold copies of data from main memory. Loading from main memory is slow the first time; subsequent access is handled by caching.

### Locality

- **Temporal locality** — if data is accessed, it's likely to be accessed again soon
- **Spatial locality** — if data is accessed, nearby data is likely to be accessed soon

### Cache coherence

In a multicore system: CPU 1 loads data D from address A and updates it to D'. If the process moves to CPU 2, CPU 2's cache might still have the old D — main memory updates are slower.

**Bus snooping** maintains a shared memory view. Each cache monitors memory updates on the bus connecting to main memory. When a CPU sees an update for a data item in its cache, it invalidates its copy or updates it (write-back cache).

**Concurrency:** sharing leads to concurrency issues → locks → hurts performance.

### Cache affinity

A process running on a specific CPU builds up state in that CPU's caches and TLBs. It's advantageous to keep running the process on the **same CPU**.

## SQMS — Single-Queue Multiprocessor Scheduling

- One queue for all CPUs
- Tries to maintain CPU affinity for jobs
- Only some jobs bounce between CPUs
- **Downside:** locking overhead on the single queue can hurt performance

## MQMS — Multi-Queue Multiprocessor Scheduling

- Multiple queues to tackle synchronization and affinity issues
- **New issue:** load imbalance — some queues busier than others
- **Fix:** **work stealing** — an idle or underutilized CPU spies on other queues, steals a job, and balances the load
