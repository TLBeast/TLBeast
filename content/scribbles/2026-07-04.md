## The idea

**MLFQ** tries to optimize both **turnaround time** and **response time**.

The scheduler doesn't know a process's characteristics upfront (like job length), so it builds a model that **learns from past behavior** to predict the future.

## Structure

- Multiple distinct queues, each with a different **priority level**
- If multiple jobs are in the same priority queue, they run in **Round Robin**

## Setting priorities

Priorities aren't fixed — they're based on observed behavior:

- **High priority** — jobs that repeatedly relinquish the CPU (e.g. interactive jobs waiting for keyboard input)
- **Low priority** — jobs with intense CPU usage over a long period

## The starvation problem

Lower-priority processes might **never run** if there are too many high-priority jobs.

## Refining the algorithm

### Attempt 1

- New jobs go in the **top queue**
- If a job uses its **full time slice**, its priority is reduced
- If a job gives up the CPU **before** its time slice is up, it stays at the same priority

This lets MLFQ approximate **SJF** for short/interactive jobs.

**Problems:**

1. **Starvation** — too many interactive jobs can consume all CPU time; long-running processes starve
2. **Gaming the system** — a program can hack the scheduler by doing I/O right before the time slice ends to keep its high priority

### Attempt 2 — priority boosting

To avoid starvation, periodically **boost the priority of all jobs**.

After time period **S**, all jobs move to the top queue. The value of S must be tuned.

### Attempt 3 — preventing hacking

Track the **total time** a process spends at a given level.

Once it crosses a threshold at that level, demote it — even if it relinquished the CPU during individual time slices.

## The five rules

1. If `priority(A) > priority(B)`, A runs and B does not
2. If `priority(A) = priority(B)`, A and B run in Round Robin
3. When a job enters the system, place it at the **highest priority**
4. Once a job uses up its time allotment at a given level (regardless of how many times it gave up the CPU), **reduce its priority**
5. After time period **S**, move **all jobs** to the topmost queue
