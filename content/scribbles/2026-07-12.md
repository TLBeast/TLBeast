## From one allocation to many

Continuing from last time — after one allocation on the heap, imagine **three** 100-byte regions allocated, with a remaining free chunk at the head:

```
[size: 100]  ← allocated
[size: 100]  ← allocated
[size: 100]  ← allocated
[size: 3764] ← free at head
```

What happens when the calling program returns some memory through `free()`?

## Freeing onto the free list

When a 100-byte region is freed, it gets **added to the free list** — inserted at the **head**:

```
head → [size: 100, next: …] → [size: 3764, next: 0]
```

One allocated chunk still in use; the free list now has **more than one element**. The free space is **fragmented**.

## Coalescing

Assume the last two in-use chunks are also freed.

If you **don't coalesce**, memory stays fragmented — lots of small free pieces that can't satisfy a bigger request.

**Solution:** merge neighboring free chunks back together so the heap becomes whole again.

```
merge neighboring chunks → one larger free region
```

## What if the heap runs out of space?

Returning `NULL` is an option.

Traditional allocators usually **start small** and request more memory from the OS with **system calls** when they need to grow.
