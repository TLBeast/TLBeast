## Assumptions (for now)

To keep things simple, we're starting with jobs where:

- Same run times
- Same arrival time
- Jobs run to completion
- Only use the CPU (no I/O)
- Run time is known beforehand

These assumptions won't last — but they help us build up the policies.

## Metric: turnaround time

**Turnaround time** = completion time − arrival time

### FIFO (First-In, First-Out)

Jobs A, B, C arrive; each takes 10s.

Avg turnaround = (10 + 20 + 30) / 3 = **20**

**Convoy effect:** if A = 100s instead, avg turnaround = (100 + 110 + 120) / 3 = **110**. Lightweights get stuck behind a heavyweight process.

### SJF (Shortest Job First)

Fixes the convoy effect — run the shortest job first.

Avg turnaround = (10 + 20 + 120) / 3 = **50**

**Problem: non-preemption.** If B and C arrive slightly later (at T = 10) while long job A is already running:

(100 + (110 − 10) + (120 − 10)) / 3 = **103.33**

### STCF (Shortest Time-to-Completion First)

Preemptive version of SJF — always run whichever job has the least time left.

((120 − 0) + (20 − 10) + (30 − 10)) / 3 = **50**

STCF is "perfect" if job lengths are known, only CPU is used, and **turnaround time** is the only metric.

## Metric: response time

Interactive time-shared machines need a different metric.

**Response time** = time of first run − arrival time

Example: A, B, C arrive at T = 0. If A and B start immediately and C waits 10s:

(0 + 0 + 10) / 3 = **3.33**

Users don't want to sit at a terminal waiting for a response because a different job got scheduled. SJF/STCF aren't good enough for response time.

## Round Robin (RR)

Uses **time slices** to improve response time.

Example: time slice of 1s vs SJF for jobs A, B, C (5s each):

| Policy | Avg response time |
|---|---|
| RR | (0 + 1 + 2) / 3 = **1** |
| SJF | (0 + 5 + 10) / 3 = **5** |

**Problem: context switching overhead.** If time slices are too short, the cost of switching between processes becomes too high.

**Trade-off:** RR makes average turnaround time suck again.
